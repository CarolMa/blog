介绍base::Thread的启动流程MessagePump 解析PostTask的原理

介绍

rhono_framework的MessagePumpYunOS的结构换过两次，一次采用了libuv，一次采用了yunos::Looper。采用libuv能够完成基本线程功能，但是在基于libuv的线程中调用dbus相关，就会crash，所以后续换成了yunos::Looper。

下面介绍下线程模型的技术细节。

base::Thread的启动流程

大家都这样用过，Thread::PostTask(Thread::SYNC, FROM_HERE, callback)，原理是什么呢。

首先来看看线程的启动流程。

很简单，在起线程的时候，调用pthread_create，然后在启动的线程里面调用了ThreadMain，ThreadMain代码如下：

 
xxxxxxxxxx
void Thread::ThreadMain() {
  ...
  scoped_ptr<MessageLoop> message_loop(message_loop_);
  message_loop_->BindToCurrentThread();
  message_loop_->set_thread_name(name_);
  message_loop_->SetTimerSlack(message_loop_timer_slack_);
  ...
  Run(message_loop_);
 ...
}

在里面调用了Run(message_loop_)，这个代码如下：

 
xxxxxxxxxx
void Thread::Run(MessageLoop* message_loop) {
  message_loop->Run();
}

实际上调用了MessageLoop::Run()，MessageLoop::Run()代码如下：

 
xxxxxxxxxx
void MessageLoop::Run() {
  DCHECK(pump_);
  RunLoop run_loop;
  run_loop.Run();
}

里面调用了base::RunLoop::Run()，实际上base::RunLoop::Run经过一些列调用，最终调用了MessagePump::Run。

MessagePump 解析

MessagePump和平台有关，可以在chromium的message_loop文件夹下，看到message_pump_default，message_pump_android，message_pump_libevent等，就是根据不同的平台、不同的编译器等来实现的。

实际上message_pump只需要以下几个能力：

  • condition_wait，条件等待，作用是：如果没有收到信号，就一直等待，如果收到信号，就破开等待。
  • singal，信号量通知，作用是：发出一个信号，将condition_wait的等待破开。
  • condition_timewait，作用是：等待一段时间，如果一段时间没有收到信号或者收到信号，就破开等待，

依赖系统的实现，所以需要各种不同的messag_pump来完成。

几乎所有的message_pump实现逻辑都类似，以MessagePumpDefault为例，简化版（去掉了一些条件判断）MessagePumpDefault::Run代码如下：

 
xxxxxxxxxx
void MessagePumpDefault::Run(Delegate* delegate) {
  ...
  for (;;) {
    bool did_work = delegate->DoWork();
    did_work |= delegate->DoDelayedWork(&delayed_work_time_);
    did_work = delegate->DoIdleWork();
    if (delayed_work_time_.is_null()) {
      event_.Wait();
    } else {
      TimeDelta delay = delayed_work_time_ - TimeTicks::Now();
      if (delay > TimeDelta()) {
        event_.TimedWait(delay);
      } else {
        delayed_work_time_ = TimeTicks();
      }
    }
  }
  ...
}

可以看到，在Run里面有一个死循环，deletegate实际上是MessageLoop，会依次调用MessageLoop::DoWork()，MessageLoop::DoDelayedWork()和DoIdelWork()。

MessageLoop::DoWork()，作用是从任务队列中取出所有的任务，如果任务能够执行（时间到了、及时任务、空闲着），就执行所有可以执行的任务，如果是delay任务，就把任务放到delay_priority_queue。

代码如下：

 
xxxxxxxxxx
bool MessageLoop::DoWork() {
  ...
  for (;;) {
    ReloadWorkQueue();
    ...
    do {
      PendingTask pending_task = work_queue_.front();
      work_queue_.pop();
      if (!pending_task.delayed_run_time.is_null()) {
        // 如果任务不是实时任务，是delay任务，就将任务放到delay_priority_queue
        AddToDelayedWorkQueue(pending_task);
        // 如果当前delay_time最小的任务就是当前任务，就告诉message_pump，下一次等待的时间是delayed_work_queue_        
        if (delayed_work_queue_.top().task.Equals(pending_task.task))
          pump_->ScheduleDelayedWork(pending_task.delayed_run_time);
      } else {
        // 执行任务
        if (DeferOrRunPendingTask(pending_task))
          return true;
      }
    } while (!work_queue_.empty());
  }
  ...
}

MessageLoop::DoDelayedWork()，维护了一个以时间为维度的优先队列，作用是执行delay任务，即PostDelayedTask抛过来的任务。

代码如下:

 
xxxxxxxxxx
bool MessageLoop::DoDelayedWork(TimeTicks* next_delayed_work_time) {  
  TimeTicks next_run_time =delayed_work_queue_.top().delayed_run_time;
  if (next_run_time > recent_time_) {
    recent_time_ = TimeTicks::Now();
    if (next_run_time > recent_time_) {
      // 时间还没到，任务不能执行，继续等待
      *next_delayed_work_time = next_run_time;
      return false;
    }
  }
}

  // 时间到了，可以执行delay任务了，取出任务，去执行。

 
xxxxxxxxxx
bool MessageLoop::DoIdelWork() {
PendingTask pending_task = delayed_work_queue_.top();
  delayed_work_queue_.pop();
  // 取出下一个最短时间，放到next_delayed_work_time中
  if (!delayed_work_queue_.empty())
    *next_delayed_work_time = delayed_work_queue_.top().delayed_run_time;
  return DeferOrRunPendingTask(pending_task);
}

  MessageLoop::DoIdleWork()，作用是执行空闲任务。

did_work表示是否有任务执行，如果三个函数都没有执行任务，就会进入等待状态，如果当前没有延时任务，就会进入event.Wait(condition_wait)，如果有延时任务，就会进入event.TimedWait(condition_timewait)，等待任务信号来临。

delayed_work_time_表示优先队列第一个元素的delay time。

通过这样一个机制，就能非常完美的执行各种任务。

PostTask的原理

原理：将任务放到message_loop的任务队列中，然后通过MessagePump::ScheduleWork，发出信号量，告诉MessagePump::Run有任务来了，然后去执行任务。

MessageLoopForYunOS(libuv)的实现：

  • event_.Wait()，换成了uv_loop_run()。
  • event_.TimedWait()，换成了uv_start_timer()，然后在timer_wake_up中去执行任务。
  • event_.Signal()，换成了uv_async_init()，然后在wake_up中执行任务。

MessageLoopForYunOS(yunos::Looper)的实现:

  • event_.Wait()，换成了yunos::Looper::run()
  • event_.TimedWait()，换成了yunos::Looper::sendDelayedTask，然后在task.Run中去执行任务。
  • event_.Singal()，换成了yunos::Looper::sendTask，然后在task.Run中执行任务。

实际上yunos::Looper中已经有了一套完整的任务机制，但是为了兼容chromium中的线程模型，这里只将yunos::Looper当做condition_variable来用。

第一版是采用了libuv实现的，能够支持在将任务抛到js线程中。

后面换成yunos::Looper，原因是很多系统接口都调用了yunos::Looper，比如dbus，以dbus的DProxy.cpp的为例：

 
xxxxxxxxxx
DProxy::DProxy(const SharedPtr<DService>& service, const String& path,
               const String& interface, bool awareLifecycle, bool reverse) {
  ...
  mQuitListener = YUNOS_NEW(QuitListener);
  if (mQuitListener.pointer()) {
    Looper::current()->addQuitListener(SharedPtr((LooperQuitListener\*) mQuitListener.pointer()));
  } else {
    YUNOS_HANDLE_OOM();
  }
  ...
}

在红色的那一行代码，调用了Looper::current()，Looper::current()中有一个tls变量，和线程有关，存储了Looper的实例变量指针，在Looper::Looper中赋值。

所以如果需要调用DBus，一定要在yunos::Looper中调用才可以，否则Looper::current()取出来的就是空，基于这一点，所以将libuv版本改成了yunos::Looper。
